# Acceptance Test Suite — UC-04 Register User Account

## Overview
**Use Case:** UC-04 Register User Account  
**Primary Actor:** Registered User (new user)  
**Objective:** Verify a new user can create an account with a valid, unique email and acceptable password, and that all alternative paths behave correctly.

---

## Assumptions / Test Data
- Supported browsers: Chrome, Firefox (latest stable).
- Base URL: `https://<cms-host>/`
- Registration page is reachable via navigation or direct route (e.g., `/register`).
- The system enforces:
  - Valid email format
  - Password security requirements (as defined by the CMS)
  - Email uniqueness
- Email Validation Service is available for format validation (or equivalent internal validation).

### Seed Data Sets
**DS-01 (Valid new user)**
- Email: `newuser_<timestamp>@example.com`
- Password: `ValidPass!234` (must satisfy CMS security rules)

**DS-02 (Existing user)**
- Email: `existing_user@example.com` (already stored in DB)
- Password: any valid password

**DS-03 (Invalid email formats)**
- `plainaddress`
- `user@`
- `@domain.com`
- `user@domain`
- `user@domain..com`

**DS-04 (Invalid passwords)**
- Too short (e.g., `Ab1!`)
- Missing required character class (e.g., no digit / no special / etc. depending on policy)

---

## UC-04-AT-01 — Registration link is available from homepage
**Priority:** P0  
**Preconditions:** CMS deployed and reachable.  
**Steps:**
1. Open the CMS homepage.
2. Locate the “Register” option.
3. Click “Register”.
**Expected Results:**
- Homepage loads successfully (HTTP 200).
- “Register” option is visible.
- Clicking “Register” navigates to the registration form.

---

## UC-04-AT-02 — Registration form displays required fields
**Priority:** P0  
**Preconditions:** Registration page is accessible.  
**Steps:**
1. Navigate to the registration page.
2. Inspect the form fields.
**Expected Results:**
- Registration form is displayed.
- Required fields include at least:
  - Email
  - Password
- A submission control exists (e.g., “Create Account”, “Register”).

---

## UC-04-AT-03 — Successful registration redirects to login (Main Success Path)
**Priority:** P0  
**Test Data:** DS-01  
**Preconditions:** Email is not already registered.  
**Steps:**
1. Navigate to the registration form.
2. Enter DS-01 email and DS-01 password.
3. Submit the form.
**Expected Results:**
- CMS validates the inputs.
- CMS creates a new user record in the database.
- CMS redirects the user to the login page.
- A success indicator is shown (message or visible login page state).

---

## UC-04-AT-04 — New account is persisted in the database
**Priority:** P1  
**Test Data:** DS-01  
**Preconditions:** Ability to verify persistence (admin DB view, API, or attempt login).  
**Steps:**
1. Complete UC-04-AT-03.
2. Attempt to log in using the newly created credentials.
**Expected Results:**
- Login succeeds using the new email/password (confirms persistence).
- (If login is out of scope here, acceptable alternative: verify record exists via DB/admin tools.)

---

## UC-04-AT-05 — Invalid email format is rejected (Extension 6a)
**Priority:** P0  
**Test Data:** DS-03  
**Preconditions:** Registration form accessible.  
**Steps (repeat for each invalid email in DS-03):**
1. Enter invalid email.
2. Enter a valid password.
3. Submit the form.
**Expected Results:**
- CMS displays an error message indicating email format is invalid.
- No user account is created.
- User remains on registration form (or returns to it) with correction requested.

---

## UC-04-AT-06 — Password that fails security standards is rejected (Extension 6b)
**Priority:** P0  
**Test Data:** DS-04  
**Preconditions:** Registration form accessible.  
**Steps (repeat for each invalid password in DS-04):**
1. Enter a valid, unique email.
2. Enter an invalid password from DS-04.
3. Submit the form.
**Expected Results:**
- CMS displays an error message indicating password requirements are not met.
- No account is created.
- User remains on registration form.

---

## UC-04-AT-07 — Duplicate email is rejected (Extension 7a)
**Priority:** P0  
**Test Data:** DS-02  
**Preconditions:** `existing_user@example.com` exists in DB.  
**Steps:**
1. Navigate to registration form.
2. Enter DS-02 email.
3. Enter a valid password.
4. Submit the form.
**Expected Results:**
- CMS displays an error message stating the email is already registered.
- No new account is created.
- User remains unregistered.

---

## UC-04-AT-08 — Database unavailable blocks account creation (Extension 8a)
**Priority:** P0  
**Test Data:** DS-01  
**Preconditions:** Simulate DB outage/unavailability (stop DB or force connection failure).  
**Steps:**
1. Navigate to registration form.
2. Enter DS-01 email/password.
3. Submit the form.
**Expected Results:**
- CMS displays a user-friendly error message indicating account creation failed.
- No sensitive technical details exposed (no stack traces, SQL, credentials).
- Account is not created.
- User remains on registration form (or error page with retry path).

---

## UC-04-AT-09 — Web server unavailable behavior (Extension 1a)
**Priority:** P0  
**Preconditions:** Simulate CMS web server down/unreachable.  
**Steps:**
1. Attempt to load homepage or registration page.
**Expected Results:**
- User receives unavailability response (browser error or CMS error page).
- If CMS error page is shown, it indicates temporary service unavailability.

---

## UC-04-AT-10 — Registration does not require login (sanity)
**Priority:** P2  
**Preconditions:** User is logged out (guest state).  
**Steps:**
1. Navigate to registration form.
2. Confirm form is accessible without authentication.
**Expected Results:**
- Registration form is reachable and usable without being logged in.

---

## UC-04-AT-11 — Cross-browser check (Chrome & Firefox)
**Priority:** P1  
**Test Data:** DS-01  
**Preconditions:** None.  
**Steps:**
1. Run UC-04-AT-03 in Chrome.
2. Run UC-04-AT-03 in Firefox (use a different unique email).
**Expected Results:**
- Registration succeeds in both browsers.
- Form validation and redirect behavior is consistent.
