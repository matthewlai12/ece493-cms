# Acceptance Test Suite — UC-28 Receive Payment Confirmation Ticket

## Overview
**Use Case:** UC-28 Receive Payment Confirmation Ticket  
**Primary Actor:** Attendee  
**Objective:** Verify that after a successful payment the CMS generates a confirmation ticket, stores it, delivers it via notification, and allows the attendee to view/save it from within the CMS. Ensure failure paths are handled safely.

---

## Assumptions / Test Data
- Supported browsers: Chrome, Firefox (latest stable).
- A successful payment already occurred (UC-27 completed).
- A “confirmation ticket” includes enough fields to serve as proof of registration (e.g., attendee name/email, conference name, payment status, confirmation/ticket ID, timestamp).
- Ticket is retrievable from the attendee’s registration details page.

### Seed Account / State
**ATT-01 Attendee**
- Email: `attendee_user@example.com`
- Password: `ValidPass!123`
- Registration status: **Confirmed (Paid)**

### Ticket Data
- Ticket ID format: `TICKET-<unique>` (example: `TICKET-88421`)
- Stored in CMS database and associated to ATT-01

---

## UC-28-AT-01 — Ticket is generated after successful payment (Main Success Path)
**Priority:** P0  
**Preconditions:** Payment recorded as successful for ATT-01.  
**Steps:**
1. Complete a successful payment for ATT-01 (or use seeded paid registration).
2. Check attendee registration record in CMS (admin view or system logs if available).
**Expected Results:**
- CMS generates a confirmation ticket.
- Ticket has a unique identifier.
- Ticket is associated with ATT-01 and marked as valid/paid.

---

## UC-28-AT-02 — Ticket is stored in database and persists
**Priority:** P0  
**Preconditions:** Ticket generated.  
**Steps:**
1. Log in as attendee (ATT-01).
2. Navigate to registration details.
3. Log out and log back in.
4. Navigate to registration details again.
**Expected Results:**
- Ticket is displayed consistently across sessions.
- Ticket content matches the latest confirmed payment status.

---

## UC-28-AT-03 — Ticket is delivered via notification (Main Success Path)
**Priority:** P0  
**Preconditions:** Notification service operational; ticket generated.  
**Steps:**
1. Complete successful payment (or use seeded paid state).
2. Check attendee email/in-app notifications.
**Expected Results:**
- Attendee receives a notification containing the ticket or a link to view it.
- Notification references successful registration/payment.

---

## UC-28-AT-04 — Attendee can view ticket within CMS (Main Success Path)
**Priority:** P0  
**Preconditions:** Ticket exists for ATT-01.  
**Steps:**
1. Log in as attendee.
2. Navigate to registration details.
**Expected Results:**
- CMS displays the payment confirmation ticket.
- Ticket is readable and clearly indicates registration is confirmed.

---

## UC-28-AT-05 — Notification delivery fails but ticket remains accessible (Extension 4a)
**Priority:** P1  
**Preconditions:** Simulate notification service outage; ticket generation still possible.  
**Steps:**
1. Complete successful payment.
2. Confirm notification fails.
3. Log in as attendee and navigate to registration details.
**Expected Results:**
- CMS logs notification failure.
- Ticket is still stored.
- Attendee can access ticket directly through CMS.

---

## UC-28-AT-06 — Ticket generation fails (Extension 2a)
**Priority:** P0  
**Preconditions:** Simulate ticket generator failure (service error/exception).  
**Steps:**
1. Record a successful payment event.
2. Trigger ticket generation.
**Expected Results:**
- CMS displays or logs an error indicating ticket generation failed.
- Ticket is not stored in the database.
- Ticket is not sent via notification.
- Registration may remain confirmed, but “ticket not available” is shown (consistent system behavior required).

---

## UC-28-AT-07 — Database unavailable prevents storing ticket (Extension 3a)
**Priority:** P0  
**Preconditions:** Simulate CMS database outage at ticket storage step.  
**Steps:**
1. Trigger ticket generation after successful payment.
**Expected Results:**
- CMS displays error indicating ticket could not be stored.
- Ticket is not retrievable later.
- No notification containing the ticket is sent (or notification contains a safe failure message/link that does not break).

---

## UC-28-AT-08 — Ticket retrieval fails / ticket unavailable (Extension 7a)
**Priority:** P0  
**Preconditions:** Ticket record missing/corrupted OR storage retrieval fails.  
**Steps:**
1. Log in as attendee.
2. Navigate to registration details.
**Expected Results:**
- CMS displays an error indicating ticket cannot be retrieved at this time.
- CMS does not display incorrect/partial ticket data.

---

## UC-28-AT-09 — Attendee not authenticated (Extension 5a)
**Priority:** P0  
**Preconditions:** Attendee logged out.  
**Steps:**
1. Attempt to access ticket/registration details page directly by URL.
**Expected Results:**
- CMS prompts attendee to log in.
- Ticket is not shown without authentication.

---

## UC-28-AT-10 — Web server unavailable behavior (Extension 1a)
**Priority:** P0  
**Preconditions:** Simulate CMS web server outage.  
**Steps:**
1. Attempt to view registration details/ticket page.
**Expected Results:**
- CMS displays system error page.
- Ticket is not displayed.

---

## UC-28-AT-11 — Ticket content correctness (proof-of-registration fields)
**Priority:** P1  
**Preconditions:** Ticket displayed.  
**Steps:**
1. Open ticket view.
2. Verify displayed fields.
**Expected Results:**
- Ticket includes:
  - Attendee identity (name/email or account identifier)
  - Conference identifier/name
  - Payment status (Paid/Confirmed)
  - Ticket/confirmation ID
  - Timestamp (payment or issuance)
- Values match registration/payment records in CMS.
