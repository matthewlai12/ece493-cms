# Use Case UC-18: Assign Referees to Submitted Papers

**Goal in Context**:  
Enable an editor to assign appropriate referees to submitted papers so that each paper undergoes the peer review process.

**Scope**:  
Conference Management System (CMS)

**Level**:  
User Goal

**Primary Actor**:  
Editor

**Secondary Actors**:  
Referee  
CMS Web Interface  
CMS Database

**Trigger**:  
The editor selects a submitted paper and initiates the referee assignment process.

---

## Success End Condition
* One or more referees are successfully assigned to the submitted paper, and review invitations are issued.

## Failed End Condition
* Referees are not assigned to the paper, and the paper remains without sufficient reviewers.

---

## Preconditions
* The editor is registered and authenticated in the CMS.
* At least one paper has been submitted and is awaiting reviewer assignment.
* A list of available referees exists in the CMS.
* The CMS referee assignment functionality is available.

---

## Main Success Scenario
1. The editor logs into the CMS.
2. The editor navigates to the list of submitted papers.
3. The editor selects a paper that requires referee assignment.
4. The CMS displays the referee assignment interface.
5. The editor selects one or more referees from the available referee list.
6. The CMS checks that each selected referee has not exceeded the maximum review workload.
7. The CMS assigns the selected referees to the paper.
8. The CMS records the assignments in the database.
9. The CMS sends review invitation notifications to the assigned referees.
10. The CMS confirms to the editor that referees have been successfully assigned.

---

## Extensions

* **6a**: Selected referee has reached the maximum review workload  
    * **6a1**: The CMS prevents assignment of that referee and informs the editor of the workload limit.

* **6b**: Assigning the referee would exceed the maximum number of referees per paper  
    * **6b1**: The CMS informs the editor that the paper already has the maximum allowed reviewers and blocks further assignments.

* **5a**: Editor selects an invalid or unavailable referee  
    * **5a1**: The CMS displays an error message and requests a valid referee selection.

* **8a**: CMS database is unavailable  
    * **8a1**: The CMS displays an error message indicating that referee assignments could not be saved.

* **1a**: CMS web server is unavailable  
    * **1a1**: The CMS displays a system error page indicating temporary service unavailability.

---

## Related Information
* **Priority**: High  
* **Frequency**: Frequent during the review assignment phase  
* **Open Issues**:  
  * Whether editors can manually override referee workload limits  
  * Whether referees should be automatically suggested based on expertise
