# Acceptance Test Suite — UC-27 Pay Conference Registration Fee

## Overview
**Use Case:** UC-27 Pay Conference Registration Fee  
**Primary Actor:** Attendee  
**Objective:** Verify that an attendee can successfully pay the conference registration fee via an external payment system, that payment confirmation is recorded, registration is marked confirmed, and notifications are sent. Ensure all failure and exception paths are handled safely.

---

## Assumptions / Test Data
- Supported browsers: Chrome, Firefox (latest stable).
- Attendee must have **initiated registration** (status: Pending Payment).
- Payment is processed by an external payment gateway (test/sandbox mode).
- A successful payment results in:
  - Payment confirmation recorded in CMS database
  - Registration status updated to Confirmed
  - Confirmation notification sent to attendee

## UC-27-AT-01 — Attendee can access payment page with correct fee details (Main Success Path)
**Priority:** P0  
**Preconditions:** Attendee authenticated; registration status is Pending Payment.  
**Steps:**
1. Log in as attendee (ATT-01).
2. Navigate to conference registration payment page.
**Expected Results:**
- CMS displays payment interface.
- Registration fee details (amount, conference name) are visible.
- “Submit Payment” action is available.

---

## UC-27-AT-02 — Successful payment confirms registration (Main Success Path)
**Priority:** P0  
**Preconditions:** Payment system available; attendee on payment page.  
**Steps:**
1. Enter valid payment information (sandbox/test card).
2. Submit payment.
**Expected Results:**
- CMS sends payment details to external payment system.
- Payment system processes transaction successfully.
- CMS records payment confirmation in database.
- CMS updates attendee registration status to **Confirmed**.
- CMS displays success confirmation to attendee.

---

## UC-27-AT-03 — Confirmation notification is sent after successful payment (Main Success Path)
**Priority:** P0  
**Preconditions:** UC-27-AT-02 completed successfully.  
**Steps:**
1. Check attendee notification channel (email/in-app).
**Expected Results:**
- Attendee receives payment confirmation notification.
- Notification references the conference and indicates registration is confirmed.

---

## UC-27-AT-04 — Payment declined by payment system (Extension 7a)
**Priority:** P0  
**Preconditions:** Payment system available; use a test card that returns “declined”.  
**Steps:**
1. Enter payment info that triggers a decline.
2. Submit payment.
**Expected Results:**
- CMS displays payment failure message.
- CMS allows attendee to retry or cancel.
- Registration remains **Unconfirmed / Pending Payment**.
- No payment confirmation is stored.

---

## UC-27-AT-05 — Payment system unavailable (Extension 6a)
**Priority:** P0  
**Preconditions:** Simulate payment gateway outage/timeouts.  
**Steps:**
1. Attempt to submit valid payment information.
**Expected Results:**
- CMS displays an error indicating payment cannot be processed at this time.
- Registration remains pending/unconfirmed.
- No payment confirmation is stored.

---

## UC-27-AT-06 — Invalid or incomplete payment information (Extension 4a)
**Priority:** P0  
**Preconditions:** Attendee on payment page.  
**Steps:**
1. Enter invalid/incomplete payment fields (e.g., missing card number or invalid expiry).
2. Submit payment.
**Expected Results:**
- CMS displays validation errors.
- CMS does not submit transaction to external payment system.
- Registration remains pending/unconfirmed.

---

## UC-27-AT-07 — Attendee cancels payment (Extension 5a)
**Priority:** P1  
**Preconditions:** Attendee on payment page.  
**Steps:**
1. Select “Cancel” on payment page.
**Expected Results:**
- CMS aborts payment workflow.
- Registration remains **Pending Payment / Unconfirmed**.
- Attendee is returned to dashboard or registration page.
- No payment record is created.

---

## UC-27-AT-08 — Database unavailable after payment success (Extension 8a)
**Priority:** P0  
**Preconditions:** Simulate CMS database outage at the moment of recording confirmation.  
**Steps:**
1. Submit a payment that external gateway reports as successful.
**Expected Results:**
- CMS displays an error indicating confirmation could not be recorded.
- CMS does not mark registration as confirmed.
- No confirmation notification is sent (since CMS cannot confirm state).
- System should not show “Confirmed” status on reload.

---

## UC-27-AT-09 — Web server unavailable behavior (Extension 1a)
**Priority:** P0  
**Preconditions:** Simulate CMS web server outage.  
**Steps:**
1. Attempt to access payment page or submit payment.
**Expected Results:**
- CMS displays system error page.
- No payment submission occurs.

---

## UC-27-AT-10 — Payment idempotency / duplicate submit protection
**Priority:** P1  
**Preconditions:** Payment submission is in progress or completed.  
**Steps:**
1. Click “Submit Payment” multiple times rapidly OR refresh during callback.
2. Return to dashboard and check registration status.
**Expected Results:**
- CMS prevents duplicate charges (only one successful payment recorded).
- Registration remains **Confirmed** after the first success.
- No duplicate payment records are created.

---

## UC-27-AT-11 — Security: only authenticated attendee in pending state can pay
**Priority:** P0  
**Preconditions:** User is logged out OR registration not initiated.  
**Steps:**
1. Attempt to access payment page directly by URL.
**Expected Results:**
- If not authenticated: CMS prompts login.
- If authenticated but not pending payment: CMS blocks access and guides user to registration flow.
- No payment interface is exposed in an invalid state.
