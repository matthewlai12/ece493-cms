# Acceptance Test Suite — UC-01 View Public Announcements

## Overview
**Use Case:** UC-01 View Public Announcements  
**Primary Actor:** Public User (Guest)  
**Objective:** Verify guests can view public announcements without authentication and that all alternative paths behave correctly.

---

## Assumptions / Test Data
- Supported browsers: Chrome, Firefox (latest stable).
- Base URL: `https://<cms-host>/`
- Announcements are stored in the CMS database and exposed through the announcements page/section.
- An “Announcements” link exists on the homepage or main navigation.

### Seed Data Sets
**DS-01 (Normal announcements set)**
- A1: Title "Registration Opens", Body present, Published date present
- A2: Title "Keynote Speaker Announced", Body present, Published date present
- A3: Title "Submission Deadline", Body present, Published date present

**DS-02 (No announcements set)**
- No announcement records exist.

---

## UC-01-AT-01 — Guest can access homepage without login
**Priority:** P0  
**Preconditions:** CMS deployed and reachable.  
**Steps:**
1. Open the CMS base URL in a supported browser.
**Expected Results:**
- Homepage loads successfully (HTTP 200).
- No login prompt is required to view the homepage.
- Navigation includes an “Announcements” option.

---

## UC-01-AT-02 — Guest can view announcements list (Main Success Path)
**Priority:** P0  
**Test Data:** DS-01  
**Preconditions:** Announcements exist in DB; guest is not logged in.  
**Steps:**
1. Navigate to the homepage.
2. Click “Announcements” in the navigation.
**Expected Results:**
- Announcements page loads successfully.
- Announcements are displayed in a readable format (e.g., list/cards).
- Each displayed announcement includes at minimum:
  - Title
  - Body/summary content
  - Published date/time (if defined by system)
- No authentication wall or redirect to login occurs.

---

## UC-01-AT-03 — Direct URL access to announcements works for guests
**Priority:** P1  
**Test Data:** DS-01  
**Preconditions:** Announcements page has a direct route (e.g., `/announcements`).  
**Steps:**
1. Enter the announcements URL directly in the address bar.
**Expected Results:**
- Announcements page loads and displays public announcements.
- No authentication required.

---

## UC-01-AT-04 — Announcements are retrieved from the database successfully
**Priority:** P1  
**Test Data:** DS-01  
**Preconditions:** DS-01 is in the database.  
**Steps:**
1. Navigate to Announcements page.
2. Compare displayed announcement count/titles to DS-01.
**Expected Results:**
- All seeded announcements appear.
- Titles match the stored records.
- No duplicate or missing announcements relative to DS-01.

---

## UC-01-AT-05 — Empty state when no announcements exist (Extension 3a)
**Priority:** P0  
**Test Data:** DS-02  
**Preconditions:** No announcements exist; guest is not logged in.  
**Steps:**
1. Navigate to Announcements page.
**Expected Results:**
- Page loads successfully.
- System displays a clear message such as:
  - “No announcements are currently available.”
- Page does not crash or show raw errors.

---

## UC-01-AT-06 — Database unavailable shows a friendly error (Extension 4a)
**Priority:** P0  
**Preconditions:** Simulate DB outage/unavailability (e.g., stop DB service or force DB connection failure).  
**Steps:**
1. Navigate to Announcements page as a guest.
**Expected Results:**
- CMS displays a user-friendly error message:
  - “Announcements cannot be retrieved at this time.”
- No sensitive technical details are shown (no stack traces, credentials, SQL).
- Guest remains unauthenticated (no unexpected redirects).

---

## UC-01-AT-07 — Web server unavailable shows system error page (Extension 1a)
**Priority:** P0  
**Preconditions:** Simulate web server down (service stopped / unreachable).  
**Steps:**
1. Attempt to load CMS homepage.
**Expected Results:**
- User receives a system unavailability response (browser error or CMS error page if served by fallback).
- If CMS provides an error page, it indicates temporary unavailability.

---

## UC-01-AT-08 — Announcements page does not expose restricted features to guests
**Priority:** P1  
**Test Data:** DS-01  
**Preconditions:** Guest is not logged in.  
**Steps:**
1. Navigate to Announcements page.
2. Inspect the page for controls such as “Create”, “Edit”, “Delete”, or admin-only UI.
**Expected Results:**
- No create/edit/delete controls are visible to the guest.
- Guest can only view announcements.

---

## UC-01-AT-09 — Formatting/readability baseline
**Priority:** P2  
**Test Data:** DS-01  
**Preconditions:** Guest is not logged in.  
**Steps:**
1. Navigate to Announcements page.
2. Verify announcements are readable (no overlapping text, truncated titles unusably, etc.).
**Expected Results:**
- Content is readable without layout breakage.
- Page is usable at common resolutions (desktop + mobile width).

---

## UC-01-AT-10 — Cross-browser check (Chrome & Firefox)
**Priority:** P1  
**Test Data:** DS-01  
**Preconditions:** Guest is not logged in.  
**Steps:**
1. Run UC-01-AT-02 in Chrome.
2. Run UC-01-AT-02 in Firefox.
**Expected Results:**
- Behavior and visibility are consistent across both browsers.
