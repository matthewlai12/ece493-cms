# Acceptance Test Suite — UC-20 Receive Completed Review Forms

## Overview
**Use Case:** UC-20 Receive Completed Review Forms  
**Primary Actor:** Editor  
**Objective:** Verify that completed review forms submitted by referees are correctly stored, associated with the appropriate paper, made accessible to the editor, and that all failure scenarios are handled safely.

---

## Assumptions / Test Data
- Supported browsers: Chrome, Firefox (latest stable).
- Review forms are required to be complete and valid before submission.
- Editors must be authenticated to access review forms.
- Review forms are associated with both a **paper** and a **referee**.
- Notifications may be asynchronous (e.g., email or in-app).

### Seed Accounts / Entities
**E-01 Editor**
- Email/Username: `editor_user@example.com`
- Password: `ValidPass!234`

**R-01 Referee**
- Email: `referee1@example.com`
- Assigned to: P-01

**R-02 Referee**
- Email: `referee2@example.com`
- Assigned to: P-01

### Paper Data
**P-01 Submitted Paper**
- Paper ID: `PAPER-801`
- Status: Under Review
- Assigned referees: R-01, R-02
- Reviews submitted: varies per test

---

## UC-20-AT-01 — Review form is stored and linked to paper on submission (Main Success Path)
**Priority:** P0  
**Test Data:** R-01 + P-01  
**Preconditions:** R-01 is assigned to P-01 and authenticated.  
**Steps:**
1. Log in as referee R-01.
2. Open assigned paper P-01.
3. Complete all required fields in the review form.
4. Submit the review form.
**Expected Results:**
- CMS validates the review form as complete.
- CMS stores the review form in the database.
- CMS associates the review with P-01 and R-01.
- Submission confirmation is shown to R-01.

---

## UC-20-AT-02 — Editor can view a single completed review
**Priority:** P0  
**Test Data:** E-01 + P-01 (1 review submitted)  
**Preconditions:** One valid review form exists for P-01.  
**Steps:**
1. Log in as editor E-01.
2. Navigate to submitted papers.
3. Select P-01.
**Expected Results:**
- CMS displays the completed review form.
- Review content is readable and attributed to the correct referee.
- No errors are displayed.

---

## UC-20-AT-03 — Editor can view multiple completed reviews
**Priority:** P0  
**Test Data:** E-01 + P-01 (reviews from R-01 and R-02)  
**Preconditions:** Two valid review forms exist for P-01.  
**Steps:**
1. Log in as editor.
2. Open P-01.
3. View all associated review forms.
**Expected Results:**
- CMS displays both review forms.
- Each review is clearly distinguishable (e.g., by referee or submission time).
- No review data is missing.

---

## UC-20-AT-04 — Notification is sent to editor after review submission
**Priority:** P1  
**Test Data:** R-01 + E-01 + P-01  
**Preconditions:** Notification service is operational.  
**Steps:**
1. R-01 submits a valid review for P-01.
2. Wait for notification delivery.
**Expected Results:**
- Editor receives a notification indicating a review was submitted.
- Notification references the correct paper.
- Failure to receive notification does not block access to the review (see AT-06).

---

## UC-20-AT-05 — Invalid or incomplete review form is rejected (Extension 1a)
**Priority:** P0  
**Test Data:** R-01 + P-01  
**Preconditions:** Review form template enforces required fields.  
**Steps:**
1. Log in as referee.
2. Leave required review fields blank.
3. Attempt to submit the review.
**Expected Results:**
- CMS rejects the submission.
- CMS displays validation errors.
- Review form is not stored.
- Editor is not notified.

---

## UC-20-AT-06 — Notification failure does not block editor access (Extension 4a)
**Priority:** P1  
**Preconditions:** Simulate notification service failure.  
**Steps:**
1. Referee submits a valid review.
2. Log in as editor.
3. Open the associated paper.
**Expected Results:**
- CMS logs notification failure.
- Editor can still manually view the completed review form.
- No data loss occurs.

---

## UC-20-AT-07 — Database unavailable prevents storing review (Extension 2a)
**Priority:** P0  
**Test Data:** R-01 + P-01  
**Preconditions:** Simulate CMS database outage.  
**Steps:**
1. Referee submits a completed review.
**Expected Results:**
- CMS displays an error indicating the review could not be saved.
- Review form is not stored.
- Editor is not notified.
- Referee is informed of failure.

---

## UC-20-AT-08 — Review form data is corrupted or unavailable (Extension 8a)
**Priority:** P0  
**Preconditions:** Simulate corrupted review data in DB.  
**Steps:**
1. Log in as editor.
2. Attempt to view reviews for affected paper.
**Expected Results:**
- CMS displays an error indicating review cannot be retrieved.
- CMS does not crash or display partial/invalid data.

---

## UC-20-AT-09 — Editor must be authenticated to view reviews (Extension 5a)
**Priority:** P0  
**Preconditions:** Editor is logged out or session expired.  
**Steps:**
1. Attempt to access review forms directly via URL.
**Expected Results:**
- CMS prompts editor to log in.
- Review forms are not displayed until authentication succeeds.

---

## UC-20-AT-10 — Web server unavailable behavior (Extension 1b)
**Priority:** P0  
**Preconditions:** Simulate CMS web server outage.  
**Steps:**
1. Attempt to submit or view review forms.
**Expected Results:**
- CMS displays system error page.
- No submission or viewing occurs.

---

## UC-20-AT-11 — Security: non-editor cannot access completed review forms
**Priority:** P0  
**Preconditions:** Non-editor account exists (author or referee).  
**Steps:**
1. Log in as non-editor user.
2. Attempt to access editor review view via UI or direct URL.
**Expected Results:**
- CMS denies access (403 or safe redirect).
- Review content is not exposed.
