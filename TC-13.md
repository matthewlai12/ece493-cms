# Acceptance Test Suite — UC-13 Receive Review Invitation

## Overview
**Use Case:** UC-13 Receive Review Invitation  
**Primary Actor:** Referee  
**Objective:** Verify that when an editor assigns a referee to a submitted paper, the CMS records a pending invitation and notifies the referee, and the referee can access the invitation to view paper details and respond. Verify all alternative paths behave correctly.

---

## Assumptions / Test Data
- Supported browsers: Chrome, Firefox (latest stable).
- Referee must exist as a registered user with referee role.
- Reviewer workload limit exists (max assigned papers; e.g., 5).
- Invitations can be accessed via:
  - An in-app “Invitations” area after login, and/or
  - A notification link (email).
- Even if notification fails, the invitation must still be visible in-app.

### Seed Accounts / Entities
**E-01 Editor account**
- Email/Username: `editor_user@example.com`
- Password: `ValidPass!234`
- Role: Editor

**R-01 Referee account (eligible)**
- Email/Username: `referee_user@example.com`
- Password: `ValidPass!234`
- Role: Referee
- Current assigned review count: 0–4 (below max)

**R-02 Referee account (at max workload)**
- Email/Username: `busy_referee@example.com`
- Password: `ValidPass!234`
- Role: Referee
- Current assigned review count: max (e.g., 5)

**R-03 Non-registered email**
- Email: `not_registered@example.com`

**P-01 Submitted paper**
- Paper ID: `PAPER-101`
- Status: Submitted
- Has required metadata and manuscript file accessible to reviewers after acceptance (depending on design)

---

## UC-13-AT-01 — Editor can initiate reviewer assignment from submitted paper
**Priority:** P0  
**Preconditions:** Editor is authenticated (E-01). Paper P-01 exists in Submitted state.  
**Steps:**
1. Log in as editor (E-01).
2. Navigate to submitted papers list.
3. Open paper P-01.
4. Select “Assign Reviewer/Referee”.
**Expected Results:**
- Reviewer assignment UI is available.
- Editor can enter a referee email address.

---

## UC-13-AT-02 — Successful invitation creation and notification (Main Success Path)
**Priority:** P0  
**Test Data:** E-01 + R-01 + P-01  
**Preconditions:** R-01 exists, eligible (below max), notification service available.  
**Steps:**
1. Log in as editor.
2. Assign R-01 to paper P-01 using `referee_user@example.com`.
3. Confirm the assignment.
**Expected Results:**
- CMS verifies referee exists and is eligible (below max assignments).
- CMS records a **pending review invitation** for R-01 and P-01 in the database.
- CMS sends a review invitation notification to R-01.
- Editor receives confirmation that the invitation was sent/created.

---

## UC-13-AT-03 — Referee can access pending invitation in CMS
**Priority:** P0  
**Test Data:** R-01 + (invitation from AT-02)  
**Preconditions:** Invitation exists for R-01.  
**Steps:**
1. Log in as referee (R-01).
2. Navigate to “Review Invitations” (or equivalent).
3. Open the invitation for paper P-01.
**Expected Results:**
- Invitation is visible with status “Pending”.
- Referee can view paper details available pre-acceptance (at minimum: title/abstract/authors per design).
- Referee has options to Accept or Decline (response handled in UC-14).

---

## UC-13-AT-04 — Referee workload limit blocks invitation (Extension 3a)
**Priority:** P0  
**Test Data:** E-01 + R-02 + P-01  
**Preconditions:** R-02 is at max workload; editor is authenticated.  
**Steps:**
1. Log in as editor.
2. Attempt to assign `busy_referee@example.com` to P-01.
**Expected Results:**
- CMS blocks the assignment.
- CMS informs editor that referee cannot be assigned due to workload limit.
- No pending invitation is recorded.
- No notification is sent.

---

## UC-13-AT-05 — Non-registered referee email is rejected (Extension 3b)
**Priority:** P0  
**Test Data:** E-01 + R-03 + P-01  
**Preconditions:** `not_registered@example.com` is not a registered referee.  
**Steps:**
1. Log in as editor.
2. Attempt to assign `not_registered@example.com` to P-01.
**Expected Results:**
- CMS informs editor that the referee is not registered.
- No pending invitation is recorded.
- No notification is sent.

---

## UC-13-AT-06 — Notification failure does not block in-app invitation (Extension 5a)
**Priority:** P1  
**Test Data:** E-01 + R-01 + P-01  
**Preconditions:** Simulate notification service outage during assignment.  
**Steps:**
1. Log in as editor.
2. Assign R-01 to P-01 while notifications are failing.
3. Log in as referee (R-01).
4. Navigate to invitations and open the invitation.
**Expected Results:**
- CMS logs notification failure.
- Invitation is still recorded in DB as pending.
- Referee can still view the invitation in-app and respond.

---

## UC-13-AT-07 — Database unavailable prevents recording/sending invitation (Extension 4a)
**Priority:** P0  
**Test Data:** E-01 + R-01 + P-01  
**Preconditions:** Simulate DB outage/unavailability at time of assignment.  
**Steps:**
1. Log in as editor.
2. Attempt to assign R-01 to P-01 while DB is unavailable.
**Expected Results:**
- CMS displays an error indicating the invitation cannot be recorded.
- No invitation record is created.
- No notification is sent.

---

## UC-13-AT-08 — Referee can access invitation via notification link (if supported)
**Priority:** P2  
**Test Data:** R-01 + invitation from AT-02  
**Preconditions:** Notification emails contain a link to invitation; notification delivered.  
**Steps:**
1. Open the invitation notification email.
2. Click the invitation link.
**Expected Results:**
- Referee is taken to the CMS invitation page.
- If not logged in, CMS prompts for login and then shows invitation.
- Invitation details are displayed correctly.

---

## UC-13-AT-09 — Web server unavailable behavior (Extension 1a)
**Priority:** P0  
**Preconditions:** Simulate CMS web server down/unreachable.  
**Steps:**
1. Attempt editor assignment or referee invitation viewing.
**Expected Results:**
- User receives unavailability response (browser error or CMS error page).
- If CMS error page is shown, it indicates temporary service unavailability.
- No partial invitation state is created on the client side.

---

## UC-13-AT-10 — Cross-browser invitation viewing is consistent
**Priority:** P2  
**Test Data:** R-01 + invitation exists  
**Preconditions:** None.  
**Steps:**
1. View invitation in Chrome.
2. View invitation in Firefox.
**Expected Results:**
- Invitation listing and details render consistently across browsers.
