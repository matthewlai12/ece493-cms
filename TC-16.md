# Acceptance Test Suite — UC-16 Submit Paper Review

## Overview
**Use Case:** UC-16 Submit Paper Review  
**Primary Actor:** Referee  
**Objective:** Verify an authenticated referee can complete and submit a review form for an assigned paper, that validation is enforced, the review is stored, and it becomes available to the editor. Verify cancellation, DB/server failures, and authorization checks.

---

## Assumptions / Test Data
- Supported browsers: Chrome, Firefox (latest stable).
- Referee must be authenticated.
- Review form template exists and is accessible for assigned papers.
- Only referees assigned to a paper may submit a review for that paper.
- “Available to editor” means the editor can view/download the submitted review in their editor workflow.

### Seed Accounts / Entities
**R-01 Referee (assigned paper)**
- Email/Username: `referee_user@example.com`
- Password: `ValidPass!234`
- Role: Referee

**R-02 Referee (not assigned to target paper)**
- Email/Username: `other_referee@example.com`
- Password: `ValidPass!234`
- Role: Referee

**E-01 Editor**
- Email/Username: `editor_user@example.com`
- Password: `ValidPass!234`
- Role: Editor

### Paper / Assignment Data
**P-01 Assigned paper for review**
- Paper ID: `PAPER-401`
- Assigned to: R-01
- Review template: available

**P-02 Unassigned paper (for authorization test)**
- Paper ID: `PAPER-402`
- Not assigned to R-01 (or assigned to a different referee)

### Review Form Inputs
**RF-01 Valid completed review**
- All required fields completed
- No invalid characters (per CMS rules)
- Ratings/comments provided as required

**RF-02 Invalid/incomplete review**
- One or more required fields blank OR invalid entry (e.g., forbidden characters)

## UC-16-AT-01 — Review form is accessible from assigned papers list
**Priority:** P0  
**Preconditions:** R-01 is authenticated and P-01 is assigned.  
**Steps:**
1. Log in as referee (R-01).
2. Navigate to “Assigned Papers”.
3. Select paper P-01.
4. Choose “Review / Open Review Form”.
**Expected Results:**
- CMS displays the review form template for P-01.
- Form fields and submit controls are visible.

---

## UC-16-AT-02 — Successful review submission stores review and confirms success (Main Success Path)
**Priority:** P0  
**Test Data:** R-01 + P-01 + RF-01  
**Preconditions:** Review form is open for P-01.  
**Steps:**
1. Complete the review form using RF-01 valid inputs.
2. Click “Submit Review”.
**Expected Results:**
- CMS validates form completeness successfully.
- CMS stores the review in the database linked to P-01 and R-01.
- CMS confirms successful submission to the referee (success message).
- Review is marked as submitted (status changes accordingly).

---

## UC-16-AT-03 — Submitted review is available to the editor
**Priority:** P0  
**Test Data:** E-01 + P-01 + result of AT-02  
**Preconditions:** A review has been submitted for P-01 by R-01.  
**Steps:**
1. Log in as editor (E-01).
2. Navigate to paper P-01 review management.
3. Locate submitted reviews for P-01.
4. Open/view the review submitted by R-01.
**Expected Results:**
- Editor can access the submitted review.
- Content matches what R-01 submitted.
- Review is associated with the correct paper and referee.

---

## UC-16-AT-04 — Incomplete/invalid review form is rejected (Extension 7a)
**Priority:** P0  
**Test Data:** R-01 + P-01 + RF-02  
**Preconditions:** Review form is open for P-01.  
**Steps:**
1. Fill the review form using RF-02 invalid/incomplete inputs.
2. Click “Submit Review”.
**Expected Results:**
- CMS displays validation errors and indicates which fields are invalid/missing.
- CMS does not store the review.
- Referee remains on the review form and can correct errors.

---

## UC-16-AT-05 — Referee cancels submission (Extension 6a)
**Priority:** P2  
**Preconditions:** A cancel action exists (button or confirmation modal) during submission.  
**Steps:**
1. Open review form for P-01.
2. Enter some review content.
3. Initiate submission (click “Submit”) if a confirmation step exists.
4. Click “Cancel” (or cancel confirmation).
**Expected Results:**
- CMS abandons the submission attempt.
- Referee returns to the review form (or remains on it) without saving a submitted review.
- Review status remains not submitted.

---

## UC-16-AT-06 — Database unavailable prevents saving review (Extension 8a)
**Priority:** P0  
**Test Data:** R-01 + P-01 + RF-01  
**Preconditions:** Simulate DB outage/unavailability at submission time.  
**Steps:**
1. Fill review form with valid inputs (RF-01).
2. Click “Submit Review” while DB is unavailable.
**Expected Results:**
- CMS displays a user-friendly error indicating the review could not be saved.
- No sensitive technical details exposed (no stack trace, SQL, credentials).
- Review is not stored and is not visible to the editor.

---

## UC-16-AT-07 — Authorization: cannot submit review for unassigned paper (Extension 3a)
**Priority:** P0  
**Test Data:** R-01 + P-02  
**Preconditions:** P-02 is not assigned to R-01.  
**Steps:**
1. Log in as R-01.
2. Attempt to access review form for P-02 (via direct URL or UI if visible).
3. Attempt to submit a review.
**Expected Results:**
- CMS blocks access or blocks submission.
- CMS displays an authorization error (not assigned / not permitted).
- No review is stored.

---

## UC-16-AT-08 — Unauthenticated user cannot access review form
**Priority:** P1  
**Preconditions:** User is logged out.  
**Steps:**
1. Attempt to access review form URL directly (if known).
**Expected Results:**
- CMS redirects to login or shows access denied.
- No form content is displayed to unauthenticated users.

---

## UC-16-AT-09 — Web server unavailable behavior (Extension 1a)
**Priority:** P0  
**Preconditions:** Simulate CMS web server down/unreachable.  
**Steps:**
1. Attempt to open review form and/or submit review.
**Expected Results:**
- User receives unavailability response (browser error or CMS error page).
- No submission is recorded.

---

## UC-16-AT-10 — Cross-browser review submission behavior is consistent
**Priority:** P2  
**Test Data:** R-01 + P-01 + RF-01  
**Preconditions:** None.  
**Steps:**
1. Submit a valid review in Chrome (use a fresh paper or reset state).
2. Submit a valid review in Firefox (different paper or reset state).
**Expected Results:**
- Form validation, submission success messaging, and status updates are consistent across browsers.
