# Acceptance Test Suite — UC-07 Change Password

## Overview
**Use Case:** UC-07 Change Password  
**Primary Actor:** Registered User  
**Objective:** Verify authenticated users can change their password securely, the new password is enforced on future logins, and all alternative/error paths are handled correctly.

---

## Assumptions / Test Data
- Supported browsers: Chrome, Firefox (latest stable).
- Base URL: `https://<cms-host>/`
- “Change Password” is available from account/profile/security settings.
- Password policy is enforced (length/complexity as defined by CMS).
- User must be authenticated to access change password form.

### Seed Accounts / Data Sets
**DS-01 (Active user account)**
- Email/Username: `active_user@example.com`
- Current Password: `ValidPass!234`
- Status: Active

**DS-02 (Valid new password)**
- New Password: `NewValidPass!567`

**DS-03 (Invalid new passwords)**
- Too short: `Ab1!`
- Missing required character class(es): `password123` (example; depends on policy)

**DS-04 (Wrong current password)**
- Wrong Current Password: `WrongPass!999`

---

## UC-07-AT-01 — Change Password option is accessible to logged-in user
**Priority:** P0  
**Preconditions:** User is authenticated (log in using DS-01).  
**Steps:**
1. From the dashboard, navigate to account/profile/security settings.
2. Locate “Change Password”.
3. Select “Change Password”.
**Expected Results:**
- Change password form is displayed.
- Form includes fields for:
  - Current password
  - New password
  - Confirm new password
- Submit control is present.

---

## UC-07-AT-02 — Successful password change (Main Success Path)
**Priority:** P0  
**Test Data:** DS-01 + DS-02  
**Preconditions:** User is authenticated; DS-01 credentials valid.  
**Steps:**
1. Open Change Password form.
2. Enter current password = `ValidPass!234`.
3. Enter new password = `NewValidPass!567`.
4. Confirm new password = `NewValidPass!567`.
5. Submit.
**Expected Results:**
- CMS validates the current password successfully.
- CMS validates the new password against policy.
- CMS updates password in the database.
- CMS displays a confirmation message that password was changed.

---

## UC-07-AT-03 — New password is required for future logins
**Priority:** P0  
**Test Data:** DS-01 + DS-02  
**Preconditions:** UC-07-AT-02 completed successfully.  
**Steps:**
1. Log out of the CMS.
2. Attempt login using the old password (`ValidPass!234`).
3. Attempt login using the new password (`NewValidPass!567`).
**Expected Results:**
- Login with old password fails (invalid credentials).
- Login with new password succeeds and redirects to dashboard.

---

## UC-07-AT-04 — Wrong current password prevents change
**Priority:** P0  
**Test Data:** DS-01 + DS-04 + DS-02  
**Preconditions:** User is authenticated with DS-01; password not changed yet (or reset to DS-01).  
**Steps:**
1. Open Change Password form.
2. Enter current password = `WrongPass!999`.
3. Enter new password = `NewValidPass!567`.
4. Confirm new password = `NewValidPass!567`.
5. Submit.
**Expected Results:**
- CMS rejects the request.
- CMS displays an error indicating current password is incorrect.
- Password in database remains unchanged (old password still works).

---

## UC-07-AT-05 — New password that violates policy is rejected
**Priority:** P0  
**Test Data:** DS-01 + DS-03  
**Preconditions:** User is authenticated with DS-01; password not changed yet (or reset).  
**Steps (repeat for each invalid password in DS-03):**
1. Open Change Password form.
2. Enter current password = `ValidPass!234`.
3. Enter invalid new password.
4. Confirm invalid new password.
5. Submit.
**Expected Results:**
- CMS displays password policy violation error.
- Password is not changed in the database.

---

## UC-07-AT-06 — New password and confirmation mismatch is rejected
**Priority:** P1  
**Test Data:** DS-01 + DS-02  
**Preconditions:** User is authenticated with DS-01.  
**Steps:**
1. Open Change Password form.
2. Enter current password = `ValidPass!234`.
3. Enter new password = `NewValidPass!567`.
4. Confirm new password = `NewValidPass!568` (mismatch).
5. Submit.
**Expected Results:**
- CMS displays an error indicating passwords do not match.
- Password is not changed.

---

## UC-07-AT-07 — Unauthenticated user cannot access change password form
**Priority:** P1  
**Preconditions:** User is logged out.  
**Steps:**
1. Attempt to navigate directly to the change password page/route (if known).
**Expected Results:**
- CMS redirects to login or shows access denied.
- Change password form is not accessible without authentication.

---

## UC-07-AT-08 — Database unavailable prevents password update
**Priority:** P0  
**Test Data:** DS-01 + DS-02  
**Preconditions:** User is authenticated; simulate DB outage/unavailability during submission.  
**Steps:**
1. Open Change Password form.
2. Enter valid current password and valid new password/confirmation.
3. Submit while DB is unavailable.
**Expected Results:**
- CMS displays a user-friendly error indicating password could not be updated.
- No sensitive technical details are exposed (no stack traces, SQL, credentials).
- Old password remains valid (no partial update).

---

## UC-07-AT-09 — Web server unavailable behavior
**Priority:** P0  
**Preconditions:** Simulate CMS web server down/unreachable.  
**Steps:**
1. Attempt to load the change password page (or settings page).
**Expected Results:**
- User receives unavailability response (browser error or CMS error page).
- If CMS error page is shown, it indicates temporary service unavailability.

---

## UC-07-AT-10 — Cross-browser change password behavior is consistent
**Priority:** P2  
**Test Data:** DS-01 + DS-02  
**Preconditions:** None.  
**Steps:**
1. Perform UC-07-AT-02 in Chrome (use a unique new password if needed).
2. Perform UC-07-AT-04 in Firefox.
**Expected Results:**
- Success and error behaviors are consistent across browsers.
