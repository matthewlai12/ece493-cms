# Acceptance Test Suite — UC-14 Accept or Reject Review Invitation

## Overview
**Use Case:** UC-14 Accept or Reject Review Invitation  
**Primary Actor:** Referee  
**Objective:** Verify a referee can respond to a pending invitation by accepting or rejecting, and the CMS records the response and updates assignment status appropriately. Verify all alternative paths (workload limit, reviewer-per-paper overflow warning, DB/server failure, cancel) behave correctly.

---

## Assumptions / Test Data
- Supported browsers: Chrome, Firefox (latest stable).
- Referees access invitations via an authenticated “Invitations” area.
- Reviewer workload limit exists (max assigned papers; e.g., 5).
- Each paper should have a maximum number of reviewers (e.g., 3).  
  - UC-14 extension 5b states the system **records acceptance** but **informs editor** if the limit is exceeded.

### Seed Accounts / Entities
**R-01 Referee (eligible)**
- Email/Username: `referee_user@example.com`
- Password: `ValidPass!234`
- Assigned review count: below max (e.g., 0–4)

**R-02 Referee (at max workload)**
- Email/Username: `busy_referee@example.com`
- Password: `ValidPass!234`
- Assigned review count: max (e.g., 5)

**E-01 Editor**
- Email/Username: `editor_user@example.com`
- Password: `ValidPass!234`

### Invitation / Paper Data
**P-01 Paper with pending invitation**
- Paper ID: `PAPER-201`
- Status: Submitted
- Has a pending invitation for R-01

**INV-01 Pending invitation**
- Invitation ID: `INV-201`
- Referee: R-01
- Paper: P-01
- Status: Pending

**P-02 Paper near/over reviewer limit**
- Paper ID: `PAPER-202`
- Status: Submitted
- Already has max reviewers assigned/accepted (e.g., 3), but a pending invitation exists for R-01

**INV-02 Pending invitation (limit overflow scenario)**
- Invitation ID: `INV-202`
- Referee: R-01
- Paper: P-02
- Status: Pending

## UC-14-AT-01 — Referee can open a pending invitation and see response options
**Priority:** P0  
**Preconditions:** INV-01 exists for R-01.  
**Steps:**
1. Log in as referee (R-01).
2. Navigate to “Pending Review Invitations”.
3. Select INV-01.
**Expected Results:**
- CMS displays paper details (at least basic metadata).
- CMS displays response options: Accept and Reject (and possibly Cancel/Back).

---

## UC-14-AT-02 — Accepting invitation records response and assigns paper to review list (Main Success Path — Accept)
**Priority:** P0  
**Test Data:** R-01 + INV-01  
**Preconditions:** R-01 is below workload max; INV-01 is pending.  
**Steps:**
1. Open INV-01.
2. Click “Accept”.
3. Confirm acceptance if prompted.
4. Navigate to referee’s “Assigned Papers / Review List”.
**Expected Results:**
- CMS records referee response as Accepted.
- Invitation status becomes Accepted (no longer pending).
- Paper P-01 appears in referee’s assigned review list.
- Editor can see the reviewer has accepted (visible in editor view or assignment state).

---

## UC-14-AT-03 — Rejecting invitation records response and marks invitation declined (Main Success Path — Reject)
**Priority:** P0  
**Test Data:** R-01 + INV-01 (fresh/reset invitation)  
**Preconditions:** INV-01 is pending.  
**Steps:**
1. Open INV-01.
2. Click “Reject”.
3. Confirm rejection if prompted.
4. Return to invitation list.
**Expected Results:**
- CMS records referee response as Rejected/Declined.
- Invitation status becomes Declined (no longer pending).
- Paper P-01 does not appear in referee’s assigned review list.
- CMS notifies editor of the rejection (or queues a notification/event).

---

## UC-14-AT-04 — Workload limit prevents acceptance (Extension 5a)
**Priority:** P0  
**Test Data:** R-02 + a pending invitation for R-02  
**Preconditions:** R-02 is at max workload; invitation is pending.  
**Steps:**
1. Log in as referee (R-02).
2. Open a pending invitation.
3. Attempt to “Accept”.
**Expected Results:**
- CMS prevents acceptance.
- CMS displays a message indicating the workload limit has been reached.
- Invitation remains Pending (or remains unresolved).
- Paper is not added to the referee’s review list.

---

## UC-14-AT-05 — Acceptance recorded but editor warned when reviewer-per-paper limit exceeded (Extension 5b)
**Priority:** P1  
**Test Data:** R-01 + INV-02 + E-01  
**Preconditions:** P-02 already has max reviewers; INV-02 is pending for R-01.  
**Steps:**
1. Log in as referee (R-01).
2. Open INV-02 for P-02.
3. Click “Accept”.
4. Log in as editor (E-01) and view reviewer assignments for P-02.
**Expected Results:**
- CMS records referee acceptance (invitation status becomes Accepted).
- CMS informs editor that reviewer limit has been exceeded (warning/message/flag).
- System behavior aligns with UC-14: acceptance is recorded but editor is alerted.

---

## UC-14-AT-06 — Database unavailable prevents recording response (Extension 6a)
**Priority:** P0  
**Test Data:** R-01 + INV-01  
**Preconditions:** Simulate DB outage during accept/reject action. Invitation is pending.  
**Steps:**
1. Log in as referee (R-01).
2. Open INV-01.
3. Click “Accept” (repeat with “Reject” if desired).
**Expected Results:**
- CMS displays an error message indicating the response could not be recorded.
- Invitation remains Pending (no state change).
- Paper is not added/removed from assigned list due to the failed transaction.
- No partial/duplicate records are created.

---

## UC-14-AT-07 — Referee cancels response action (Extension 5c)
**Priority:** P2  
**Preconditions:** Invitation is pending and a cancel/back action exists before final commit (or confirmation modal exists).  
**Steps:**
1. Open a pending invitation.
2. Click “Accept” or “Reject”.
3. Cancel at confirmation prompt (or click “Cancel/Back”).
**Expected Results:**
- CMS returns referee to invitation list or invitation detail view.
- No response is recorded.
- Invitation remains Pending.

---

## UC-14-AT-08 — Invitation removed from pending list after response
**Priority:** P1  
**Test Data:** R-01 + INV-01  
**Preconditions:** INV-01 pending.  
**Steps:**
1. Record a response (Accept or Reject) for INV-01.
2. Return to pending invitations list.
**Expected Results:**
- INV-01 no longer appears in the Pending list.
- It appears under Accepted/Declined history (if such views exist) or is otherwise not pending.

---

## UC-14-AT-09 — Unauthenticated referee cannot respond
**Priority:** P1  
**Preconditions:** User is logged out.  
**Steps:**
1. Attempt to open an invitation via direct URL (if known).
**Expected Results:**
- CMS redirects to login or shows access denied.
- No response can be recorded without authentication.

---

## UC-14-AT-10 — Web server unavailable behavior (Extension 1a)
**Priority:** P0  
**Preconditions:** Simulate CMS web server down/unreachable.  
**Steps:**
1. Attempt to view invitations or submit Accept/Reject.
**Expected Results:**
- User receives unavailability response (browser error or CMS error page).
- No response is recorded.

---

## UC-14-AT-11 — Cross-browser response behavior is consistent
**Priority:** P2  
**Test Data:** R-01 + INV-01  
**Preconditions:** INV-01 is pending.  
**Steps:**
1. Accept an invitation in Chrome (or reject).
2. Reset another invitation to pending (or use INV-02).
3. Reject (or accept) in Firefox.
**Expected Results:**
- Accept/Reject actions behave consistently across browsers.
- Status updates and list updates are consistent.
