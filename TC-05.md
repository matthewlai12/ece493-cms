# Acceptance Test Suite — UC-05 Validate Registration Information

## Overview
**Use Case:** UC-05 Validate Registration Information  
**Primary Actor:** Registered User (new user during registration)  
**Objective:** Verify the CMS validates registration inputs (email format, email uniqueness, password policy, required fields) and handles all alternative paths correctly before allowing account creation.

---

## Assumptions / Test Data
- Supported browsers: Chrome, Firefox (latest stable).
- Base URL: `https://<cms-host>/`
- Registration form route: `/register` (or reachable via homepage “Register”).
- Validation occurs on form submission (server-side), optionally with client-side hints.
- Required fields include at minimum: Email, Password (and any other required fields in your form such as Name).

### Seed Data Sets
**DS-01 (Valid registration input)**
- Email: `valid_user_<timestamp>@example.com`
- Password: `ValidPass!234` (meets CMS policy)
- All other required fields filled with valid values

**DS-02 (Invalid email formats)**
- `plainaddress`
- `user@`
- `@domain.com`
- `user@domain`
- `user@domain..com`

**DS-03 (Duplicate email)**
- Email: `existing_user@example.com` (already in DB)
- Password: valid password

**DS-04 (Invalid passwords)**
- Too short: `Ab1!`
- Missing policy requirement(s): examples depend on your policy (e.g., `password123`, `NoSpecial123`)

**DS-05 (Missing required fields)**
- Missing email
- Missing password
- Missing other required field(s) (e.g., name)

---

## UC-05-AT-01 — Validation succeeds for complete, valid input (Main Success Path)
**Priority:** P0  
**Test Data:** DS-01  
**Preconditions:** Registration form accessible; DS-01 email not in DB.  
**Steps:**
1. Navigate to the registration form.
2. Enter DS-01 values for all required fields.
3. Submit the registration form.
**Expected Results:**
- CMS validates inputs successfully.
- No validation errors are shown.
- CMS indicates validation success and allows progression to account creation (e.g., registration continues / account created / next step shown).

---

## UC-05-AT-02 — Invalid email format is rejected (Extension 3a)
**Priority:** P0  
**Test Data:** DS-02  
**Preconditions:** Registration form accessible.  
**Steps (repeat for each email in DS-02):**
1. Enter an invalid email from DS-02.
2. Enter a valid password and other required fields.
3. Submit the registration form.
**Expected Results:**
- CMS displays an email format error message.
- Submission does not proceed to account creation.
- User remains on the registration form (or is returned to it).
- No account is created.

---

## UC-05-AT-03 — Duplicate email is rejected (Extension 4a)
**Priority:** P0  
**Test Data:** DS-03  
**Preconditions:** `existing_user@example.com` exists in the database.  
**Steps:**
1. Enter `existing_user@example.com` and a valid password and other required fields.
2. Submit the registration form.
**Expected Results:**
- CMS displays an error stating the email is already registered.
- Submission does not proceed to account creation.
- No new account is created.

---

## UC-05-AT-04 — Password policy violations are rejected (Extension 5a)
**Priority:** P0  
**Test Data:** DS-04  
**Preconditions:** Registration form accessible; email is unique.  
**Steps (repeat for each invalid password in DS-04):**
1. Enter a valid, unique email.
2. Enter an invalid password from DS-04.
3. Fill all other required fields.
4. Submit the registration form.
**Expected Results:**
- CMS displays a password policy violation error.
- Submission does not proceed to account creation.
- No account is created.

---

## UC-05-AT-05 — Missing required fields are detected and highlighted (Extension 6a)
**Priority:** P0  
**Test Data:** DS-05  
**Preconditions:** Registration form accessible.  
**Steps (repeat for each missing-field variant in DS-05):**
1. Leave one required field blank.
2. Fill all other required fields with valid data.
3. Submit the registration form.
**Expected Results:**
- CMS highlights the missing required field(s).
- CMS displays a message requesting completion.
- Submission does not proceed to account creation.

---

## UC-05-AT-06 — Multiple validation errors are handled together (combined extensions)
**Priority:** P1  
**Preconditions:** Registration form accessible.  
**Steps:**
1. Enter an invalid email format (from DS-02).
2. Enter an invalid password (from DS-04).
3. Submit the registration form.
**Expected Results:**
- CMS reports validation failures (either multiple messages at once or the first blocking error, per implementation).
- No account is created.
- User can correct inputs and resubmit.

---

## UC-05-AT-07 — Database unavailable prevents uniqueness validation (Extension 7a)
**Priority:** P0  
**Test Data:** DS-01  
**Preconditions:** Simulate DB outage/unavailability (stop DB or force connection failure).  
**Steps:**
1. Navigate to the registration form.
2. Enter valid values (DS-01).
3. Submit the registration form.
**Expected Results:**
- CMS displays an error message indicating validation cannot be completed at this time.
- Submission does not proceed to account creation.
- No sensitive details are shown (no stack trace, SQL, credentials).

---

## UC-05-AT-08 — User cancels submission (Extension 2a)
**Priority:** P2  
**Preconditions:** Registration form accessible; a cancel action exists (Cancel button/back navigation).  
**Steps:**
1. Enter some registration information.
2. Click “Cancel” (or navigate back using provided UI).
**Expected Results:**
- CMS aborts validation/registration submission flow.
- User is returned to the registration form (or homepage) without creating an account.
- No partial account is created.

---

## UC-05-AT-09 — Validation does not create an account by itself
**Priority:** P1  
**Test Data:** DS-01  
**Preconditions:** Ability to verify DB state or login behavior.  
**Steps:**
1. Submit valid registration info.
2. If the system separates validation from creation, verify no user record exists until creation step completes.
**Expected Results:**
- If validation and creation are separate steps: no account is created at validation-only stage.
- If validation is part of the same submission that creates the account: creation occurs only after successful validation (no partial/invalid records).

---

## UC-05-AT-10 — Cross-browser validation behavior is consistent
**Priority:** P1  
**Test Data:** DS-01 + one invalid case from DS-02  
**Preconditions:** None.  
**Steps:**
1. Run UC-05-AT-01 in Chrome.
2. Run UC-05-AT-02 in Firefox.
**Expected Results:**
- Validation messages and blocking behavior are consistent across browsers.
