# Acceptance Test Suite — UC-11 Receive Paper Decision

## Overview
**Use Case:** UC-11 Receive Paper Decision  
**Primary Actor:** Author  
**Objective:** Verify that once an editor finalizes a paper decision (accept/reject), the decision is stored, the author is notified (if possible), and the author can view the decision in the CMS. Verify all alternative paths behave correctly.

---

## Assumptions / Test Data
- Supported browsers: Chrome, Firefox (latest stable).
- Authors must be authenticated to view decision details.
- Decisions exist only after required reviews are complete.
- Notification is sent via email (or equivalent), but decision must still be viewable in-app even if notification fails.

### Seed Accounts / Entities
**A-01 Author account**
- Email/Username: `author_user@example.com`
- Password: `ValidPass!234`
- Role: Author

**E-01 Editor account**
- Email/Username: `editor_user@example.com`
- Password: `ValidPass!234`
- Role: Editor

### Paper / Review Data Sets
**P-01 Paper with complete reviews**
- Paper ID: `PAPER-001`
- Owner: A-01
- Reviews: 3 completed reviews present
- Decision: not yet recorded

**P-02 Paper with incomplete reviews**
- Paper ID: `PAPER-002`
- Owner: A-01
- Reviews: fewer than required (e.g., 2 completed)
- Decision: not yet allowed

### Decision Values
**D-ACCEPT:** Accepted  
**D-REJECT:** Rejected

---

## UC-11-AT-01 — Editor can finalize decision only after all reviews complete (Extension 1a)
**Priority:** P0  
**Test Data:** E-01 + P-02  
**Preconditions:** Editor is authenticated; P-02 exists with incomplete reviews.  
**Steps:**
1. Log in as editor (E-01).
2. Navigate to the decision action for P-02.
3. Attempt to finalize a decision (accept or reject).
**Expected Results:**
- CMS blocks the finalize action.
- CMS informs editor that additional reviews are required.
- No decision record is created.
- No notification is sent to the author.

---

## UC-11-AT-02 — Finalizing decision stores record and triggers notification (Main Success Path)
**Priority:** P0  
**Test Data:** E-01 + P-01 + D-ACCEPT (repeat with D-REJECT if required)  
**Preconditions:** Editor is authenticated; P-01 exists with all required reviews complete.  
**Steps:**
1. Log in as editor (E-01).
2. Open paper P-01 decision screen.
3. Select decision = Accepted (or Rejected).
4. Confirm/finalize the decision.
**Expected Results:**
- CMS stores the decision in the database for P-01.
- CMS triggers a notification to the author indicating a decision is available.
- Editor receives confirmation that the decision was recorded.

---

## UC-11-AT-03 — Author can view decision in CMS after login (Main Success Path)
**Priority:** P0  
**Test Data:** A-01 + P-01 + decision recorded from AT-02  
**Preconditions:** A decision exists for P-01 in DB.  
**Steps:**
1. Log in as author (A-01).
2. Navigate to “My Submissions” / list of submitted papers.
3. Select paper P-01.
**Expected Results:**
- CMS displays the final decision (Accepted/Rejected) for P-01.
- Decision displayed matches what editor recorded.
- Author can view decision without errors.

---

## UC-11-AT-04 — Notification failure does not block in-app decision viewing (Extension 3a)
**Priority:** P1  
**Test Data:** E-01 + A-01 + P-01  
**Preconditions:** Simulate notification service failure (e.g., email service down) during decision finalization.  
**Steps:**
1. Log in as editor (E-01).
2. Finalize a decision for P-01.
3. Log in as author (A-01).
4. Navigate to P-01 decision view.
**Expected Results:**
- CMS logs notification failure (verifiable via admin log if available).
- Decision is still stored in DB.
- Author can still view the decision after logging in.

---

## UC-11-AT-05 — Unauthenticated author is prompted to log in (Extension 4a)
**Priority:** P1  
**Test Data:** P-01 decision exists  
**Preconditions:** User is logged out.  
**Steps:**
1. Attempt to access the decision details page directly (if route is known) OR try to access “My Submissions”.
**Expected Results:**
- CMS prompts user to log in (or redirects to login).
- Decision details are not shown to unauthenticated users.

---

## UC-11-AT-06 — Decision record unavailable shows friendly error (Extension 6a)
**Priority:** P0  
**Test Data:** A-01 + P-01  
**Preconditions:** Decision exists but retrieval fails (simulate: DB read error, record corrupted/missing, or service failure).  
**Steps:**
1. Log in as author (A-01).
2. Navigate to paper P-01 decision view.
**Expected Results:**
- CMS displays a user-friendly error indicating the decision cannot be retrieved at this time.
- No sensitive technical details are exposed (no stack trace, SQL, credentials).

---

## UC-11-AT-07 — Database unavailable prevents saving decision and prevents notification (Extension 2a)
**Priority:** P0  
**Test Data:** E-01 + P-01  
**Preconditions:** Simulate DB outage/unavailability at the moment the editor finalizes decision.  
**Steps:**
1. Log in as editor (E-01).
2. Attempt to finalize decision for P-01 while DB is unavailable.
**Expected Results:**
- CMS displays an error message indicating decision could not be stored.
- Decision is not saved.
- No notification is sent to the author (since storage did not succeed).

---

## UC-11-AT-08 — Author sees correct decision status (Accepted vs Rejected)
**Priority:** P1  
**Test Data:** A-01 + P-01  
**Preconditions:** Run twice:
- Case A: editor records Accepted
- Case B: editor records Rejected
**Steps:**
1. For each case, log in as author and open the paper decision view.
**Expected Results:**
- CMS displays the correct decision label corresponding to the saved decision.
- No ambiguity (clearly indicates Accepted or Rejected).

---

## UC-11-AT-09 — Web server unavailable behavior
**Priority:** P0  
**Preconditions:** Simulate CMS web server down/unreachable.  
**Steps:**
1. Attempt to load login, submissions page, or decision page.
**Expected Results:**
- User receives unavailability response (browser error or CMS error page).
- If CMS error page is shown, it indicates temporary service unavailability.

---

## UC-11-AT-10 — Cross-browser decision viewing is consistent
**Priority:** P2  
**Test Data:** A-01 + P-01 decision exists  
**Preconditions:** None.  
**Steps:**
1. View decision in Chrome.
2. View decision in Firefox.
**Expected Results:**
- Decision display is consistent across browsers.
