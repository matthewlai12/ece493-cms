# Use Case UC-28: Receive Payment Confirmation Ticket

**Goal in Context**:  
Provide the attendee with a confirmation ticket after successful payment so that they have verifiable proof of conference registration.

**Scope**:  
Conference Management System (CMS)

**Level**:  
User Goal

**Primary Actor**:  
Attendee

**Secondary Actors**:  
CMS Web Interface  
CMS Database  
Notification Service (e.g., email)

**Trigger**:  
The CMS records a successful conference registration payment.

---

## Success End Condition
* The attendee receives a payment confirmation ticket that can be viewed or saved as proof of registration.

## Failed End Condition
* The attendee does not receive or cannot access the payment confirmation ticket.

---

## Preconditions
* The attendee is registered and authenticated in the CMS.
* The attendee has successfully completed the conference registration payment.
* The CMS confirmation generation and notification mechanisms are available.

---

## Main Success Scenario
1. The CMS confirms that the attendee’s payment was successful.
2. The CMS generates a payment confirmation ticket.
3. The CMS stores the confirmation ticket in the database.
4. The CMS sends the confirmation ticket to the attendee via notification (e.g., email).
5. The attendee logs into the CMS.
6. The attendee navigates to their registration details.
7. The CMS displays the payment confirmation ticket.

---

## Extensions

* **4a**: Notification delivery fails  
    * **4a1**: The CMS logs the notification failure and allows the attendee to access the confirmation ticket directly through the CMS.

* **2a**: Confirmation ticket generation fails  
    * **2a1**: The CMS displays an error message and does not store or send the ticket.

* **3a**: CMS database is unavailable  
    * **3a1**: The CMS displays an error message indicating that the confirmation ticket could not be stored.

* **7a**: Confirmation ticket is unavailable  
    * **7a1**: The CMS displays an error message indicating that the ticket cannot be retrieved at this time.

* **5a**: Attendee is not authenticated  
    * **5a1**: The CMS prompts the attendee to log in before accessing the confirmation ticket.

* **1a**: CMS web server is unavailable  
    * **1a1**: The CMS displays a system error page indicating temporary service unavailability.

---

## Related Information
* **Priority**: High  
* **Frequency**: Once per attendee  
* **Open Issues**:  
  * Whether confirmation tickets should include QR codes or printable formats  
  * Whether tickets should be resent upon request
