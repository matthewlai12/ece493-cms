# Acceptance Test Suite — UC-10 Save Submission Progress

## Overview
**Use Case:** UC-10 Save Submission Progress  
**Primary Actor:** Author  
**Objective:** Verify an authenticated author can save a partially completed submission, later retrieve it without data loss, and that invalid data, cancellation, and system failures are handled correctly.

---

## Assumptions / Test Data
- Supported browsers: Chrome, Firefox (latest stable).
- Author must be authenticated to access submission features.
- “Save” creates/updates a **draft** (not a final submitted paper).
- Drafts are visible to the author (e.g., “Draft Submissions” / “My Submissions” with status Draft).

### Seed Account
**DS-01 (Valid author account)**
- Email/Username: `author_user@example.com`
- Password: `ValidPass!234`
- Role: Author

### Draft Metadata Inputs
**M-01 Partial valid draft data**
- Title: present (if applicable)
- Abstract: present (or partially filled)
- Keywords: present (optional depending on required-at-save policy)
- Contact info: valid
- Some fields intentionally left blank but still acceptable for saving progress (unless your validation requires them)

**M-02 Invalid draft data**
- Example invalid values:
  - Contact email malformed (e.g., `bademail@`)
  - Fields containing disallowed characters (if applicable)
  - Any field failing validation rules enforced at save time

---

## UC-10-AT-01 — Save option is available during submission
**Priority:** P0  
**Preconditions:** Author is logged in (DS-01) and has opened a new submission form.  
**Steps:**
1. Navigate to the paper submission form.
2. Locate the “Save” option/button.
**Expected Results:**
- “Save” option is visible and enabled.
- Author can attempt to save without submitting the paper.

---

## UC-10-AT-02 — Successful save of partial submission (Main Success Path)
**Priority:** P0  
**Test Data:** DS-01 + M-01  
**Preconditions:** Author is logged in and on submission form.  
**Steps:**
1. Enter M-01 partial data into the submission form.
2. Click “Save”.
**Expected Results:**
- CMS validates currently entered data (per rules).
- CMS stores the partial submission data in the database as a draft.
- CMS confirms that progress has been saved.
- Author remains able to continue editing or exit.

---

## UC-10-AT-03 — Saved draft can be retrieved later without data loss
**Priority:** P0  
**Test Data:** DS-01 + (result of AT-02)  
**Preconditions:** A draft exists from UC-10-AT-02.  
**Steps:**
1. Exit the submission form (navigate away or return to dashboard).
2. Re-open the saved draft (e.g., via “My Submissions” / “Drafts”).
**Expected Results:**
- Draft is visible in the author’s list of submissions with status “Draft” (or equivalent).
- Previously entered fields match what was saved.
- No unexpected data loss occurs.

---

## UC-10-AT-04 — Multiple saves update the same draft
**Priority:** P1  
**Test Data:** DS-01 + M-01  
**Preconditions:** Draft exists and is open.  
**Steps:**
1. Edit one or more fields in the draft (change abstract/keywords/etc.).
2. Click “Save” again.
3. Exit and re-open the draft.
**Expected Results:**
- CMS confirms save success.
- Draft reflects the most recent changes on re-open.
- No duplicate draft is created unintentionally (unless your design explicitly creates versions).

---

## UC-10-AT-05 — Invalid/incomplete values block saving (Extension 3a)
**Priority:** P0  
**Test Data:** DS-01 + M-02  
**Preconditions:** Author is logged in and on submission form.  
**Steps:**
1. Enter M-02 invalid data (e.g., malformed contact email).
2. Click “Save”.
**Expected Results:**
- CMS displays validation error messages.
- CMS does not save the draft changes.
- Author remains on the form and can correct errors.

---

## UC-10-AT-06 — Database unavailable prevents saving (Extension 4a)
**Priority:** P0  
**Test Data:** DS-01 + M-01  
**Preconditions:** Simulate DB outage/unavailability during save.  
**Steps:**
1. Enter M-01 partial data.
2. Click “Save” while DB is unavailable.
**Expected Results:**
- CMS displays a user-friendly error indicating progress could not be saved.
- No sensitive technical details exposed (no stack traces, SQL, credentials).
- New changes are not persisted.

---

## UC-10-AT-07 — Author cancels save operation (Extension 2a)
**Priority:** P2  
**Preconditions:** Save operation presents a confirmation dialog/modal OR user can cancel from the save flow.  
**Steps:**
1. Enter some new data into the form.
2. Initiate “Save”.
3. Cancel the save (e.g., click “Cancel” in confirmation dialog).
**Expected Results:**
- CMS abandons the save operation.
- User is returned to the submission form.
- Changes are not saved.

---

## UC-10-AT-08 — Saved draft is not treated as a final submitted paper
**Priority:** P1  
**Test Data:** DS-01 + M-01  
**Preconditions:** Draft saved successfully.  
**Steps:**
1. Go to “My Submissions”.
2. Inspect the saved draft’s state.
**Expected Results:**
- Draft is clearly not marked as “Submitted”.
- Draft does not enter the review process.
- Any “Submitted” indicators (review status, referee assignment) are absent.

---

## UC-10-AT-09 — Unauthenticated access to drafts is blocked
**Priority:** P1  
**Preconditions:** User is logged out.  
**Steps:**
1. Attempt to access a draft edit URL directly (if known).
**Expected Results:**
- CMS redirects to login or shows access denied.
- Draft content is not exposed to unauthenticated users.

---

## UC-10-AT-10 — Web server unavailable behavior (Extension 1a)
**Priority:** P0  
**Preconditions:** Simulate CMS web server down/unreachable.  
**Steps:**
1. Attempt to load submission form and/or perform save.
**Expected Results:**
- User receives unavailability response (browser error or CMS error page).
- If CMS error page is shown, it indicates temporary service unavailability.
- No save occurs.

---

## UC-10-AT-11 — Cross-browser save/retrieve behavior is consistent
**Priority:** P2  
**Test Data:** DS-01 + M-01  
**Preconditions:** None.  
**Steps:**
1. Save a draft in Chrome (UC-10-AT-02).
2. Log out.
3. Log in using Firefox.
4. Retrieve the saved draft (UC-10-AT-03).
**Expected Results:**
- Draft is accessible across browsers.
- Saved fields match expected values.
